# LLM Agent 编排架构对比示例

本项目是一份用于横向对比 **LlamaIndex Workflows**、**LangChain (LCEL)** 以及 **LangGraph** 核心设计哲学的教学示例。
针对相同的业务场景（带循环回退的客服工单分发与起草），我们使用三种市面上最流行的大模型编排架构进行了独立的代码实现。

## ⚙️ 运行与启动命令

本程序运行依赖处于项目根目录下的专属虚拟环境 (`venv`)，且需要先在 [`.env`](./.env) 文件中配置好对应的硅基流动 API 秘钥（`SILICONFLOW_API_KEY`）。

启动时，请打开终端并首先激活环境：
```bash
source venv/bin/activate
```

随后，即可独立运行任意一条链路示例：
```bash
# 1. 运行事件驱动型 LlamaIndex 方案
python llama_index_workflow.py

# 2. 运行单向声明式 LangChain LCEL 方案
python langchain_lcel.py

# 3. 运行图状态机 LangGraph 方案
python langgraph_app.py
```

## 🎯 测试场景：智能客服反思工作流

为了能暴露出各个框架的核心差异，此场景包含完整的**顺序、分支(Routing)、以及死循环(Cycle)**：
1. **分类(Classify)**：识别用户意图是账单(Billing)或技术(Tech)。
2. **起草(Drafting)**：起草相应的安抚答复。
3. **内容审查(Review/Reflection)**：对起草的文案运用 LLM 进行独立“反思验证”，检查语气必须友善。**如果不过关，将其直接打回上游重新生成**，直至通过。

## 📊 预期学习效果 (对比总结)

学习者通过研究 `llama_index_workflow.py`、`langchain_lcel.py` 和 `langgraph_app.py` 的源码差别，可以得到以下结论：

1. **LlamaIndex Workflows (事件驱动)**
   - **特点**：代码由一系列装饰器 `@step` 构成，一切转场都基于 `Event` 类的高声广播与订阅。
   - **体验**：它像极了后端的事件总线 (Event Bus)，它实现循环的方式是“下游处理失败后，朝系统继续扔一个和原来相同的事件”。极其解耦，但很难用肉眼一眼看懂执行的闭环逻辑线。

2. **Langchain LCEL (声明式管道)**
   - **特点**：代码非常优雅紧凑，以运算符 `chain = classify | draft | review` 的面貌示人。
   - **体验**：对于一直往前走的任务它所向披靡。但在面对本项目要求的 **“Review 失败要掉头重做”** 的情况下，完全抓瞎。LCEL 无法优雅地原路返回，最终只能倒退成把业务强行包在原生 Python 的 `while` 循环里，破坏了组件链式纯洁性。

3. **LangGraph (状态显式图)**
   - **特点**：将业务设计成了数学图形（边和节点）。具备最高维度的全链路状态追踪。
   - **体验**：对付“循环重试”这类 Agent 的必备功能手到擒来。通过 `add_conditional_edges` 轻松实现在审核不通过时用箭头重新指向上一步。代码稍微冗长，但是是做各种复杂 AI 智能体产品的不二之选。
